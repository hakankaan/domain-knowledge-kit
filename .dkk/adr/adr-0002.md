# ADR-0002 — Flow-Anchored Story Generation with Copilot Skill Distribution

**Status:** Accepted  
**Date:** 2026-03-27  
**Deciders:** Kaan Akın

## Context

Developers using AI assistants to write or reshape user stories face three recurring problems:

1. **Hallucinated terminology.** The AI invents entity names, commands, or events that don't exist in the domain model.
2. **Missing constraints.** Architectural decisions (ADRs) and business policies are ignored because the AI has no structured access to them.
3. **Generic acceptance criteria.** The AI writes vague criteria disconnected from the actual domain events and policies.

DKK already holds all the structured knowledge needed to solve these problems — commands with actors and preconditions, events with fields, policies with when/then rules, BDD examples with given/when/then scenarios, ADRs, and cross-context flows. However, no single command aggregated this information in a story-ready format, and no mechanism taught AI assistants how to use DKK for story workflows.

Three design questions required resolution:

1. **What domain construct should anchor story generation?** Individual commands, aggregates, or flows?
2. **How should AI assistants learn the story workflow?** A CLI command alone, a Copilot prompt file, or a Copilot skill?
3. **How should skills reach consumer projects?** Skills live in the DKK package repository — consumers installing DKK don't automatically get them.

## Decision

### 1. Flows as the story anchor

The `flow` type was chosen as the primary entry point for story generation. A flow represents an ordered, cross-context business process — it maps naturally to an Epic (large flow) or a User Story (small flow). Individual steps within the flow (commands, events, policies, read models) provide the granularity for story slicing.

The flow's existing graph wiring (`flow_step`, `flow_next` edges) and the domain model's item-level richness (actors, preconditions, rejections, examples, ADR refs) provide everything needed without schema changes.

### 2. A dedicated CLI command (`dkk story`) paired with a Copilot Skill

A new `dkk story <flow-id>` command aggregates a flow's full context — actors, ordered steps with their details, triggered policies, BDD examples, linked ADRs, and downstream read models — into a single structured output (Markdown or JSON).

A Copilot Skill (`.github/skills/story-analyst/skill.md`) was chosen over a Copilot Prompt file because:

- **Skills encode behavioral knowledge** (noun enforcement, epic splitting rules, reshaping guidance) that persists across all interactions. A prompt is a one-shot trigger with no reusable judgment.
- **Skills are portable** across AI tools (Copilot, Cursor, Codex). Prompts are Copilot-specific.
- **Skills activate contextually** when the AI detects story-related work. Prompts require explicit user selection from a menu.

A prompt would add negligible value on top of the skill — it would amount to "run `dkk story` and format the output," which the skill already covers.

### 3. `dkk init --skills` for distribution

Skills are copied from the DKK package into the consumer's `.github/skills/` directory via `dkk init --skills`. The behavior is non-destructive (existing files are skipped) with `--force` to overwrite for version upgrades. This extends the existing `dkk init` onboarding command rather than introducing a separate workflow.

## Consequences

### Positive

- **Single-command context retrieval.** `dkk story <flow-id>` replaces 5–15 serial CLI calls that an AI would otherwise need to stitch together.
- **Domain-grounded stories.** Noun enforcement in the skill prevents hallucinated terminology. Acceptance criteria are derived from actual policies and BDD examples in the domain model.
- **No schema changes.** The existing flow schema (name, description, ordered steps with ref/type/note) and the domain item schemas (commands with actors/preconditions/examples, events with fields, policies with when/then) were sufficient.
- **Portable skill.** The story-analyst skill works with any AI tool that reads `.github/skills/` or equivalent skill directories.
- **Smooth onboarding.** `dkk init --skills` gives consumers all AI tooling in one command.

### Negative

- **Flow coverage dependency.** Story generation quality depends on flows being defined in the domain model. If a team hasn't modeled flows, the skill falls back to `dkk summary` + `dkk related` on individual commands, which produces less structured output.
- **Skill maintenance.** The skill file ships with the DKK package. Changes to the `dkk story` output format require updating the skill in lockstep. `--force` mitigates this for consumers but doesn't automate it.
